# Commit

Create one focused, reviewable commit from the intended changes.

## Workflow

1. Check branch and status:

```bash
git branch --show-current
git status --short
```

2. If on `main` or `master`, create a feature branch (use the `create-branch` skill) unless the user explicitly asked to commit there.
3. Inspect the diff before staging or committing.
4. Stage only intended files; do not use `git add .` unless the user explicitly wants every change included.
5. Choose the commit type and scope from the staged diff.
6. Commit only if there are staged changes.
7. Keep the repository in a working state after the commit.

## Message Format

```text
<type>(<scope>): <Subject>

<body>

<footer>
```

Rules:

- Header is required; body and footer are optional.
- Scope is optional.
- Subject uses imperative present tense: `Add`, not `Added`.
- Subject starts with a capital letter and has no trailing period.
- Subject is at most 70 characters.
- All lines are at most 100 characters.
- Body explains what changed and why.
- Use real newlines; never put literal `\n` sequences in commit messages.

## Types

| Type      | Use for                          |
| --------- | -------------------------------- |
| `feat`    | user-facing feature              |
| `fix`     | bug fix                          |
| `ref`     | refactor without behavior change |
| `perf`    | performance improvement          |
| `docs`    | documentation                    |
| `test`    | tests                            |
| `build`   | build system or dependencies     |
| `ci`      | CI configuration                 |
| `chore`   | maintenance                      |
| `style`   | formatting only                  |
| `meta`    | repository metadata              |
| `license` | license changes                  |
| `revert`  | reverting a prior commit         |

## Footers

Use issue references only when the exact ID or URL is present in user input, branch names, commits, or verified tracker output.

```text
Fixes #1234
Fixes GH-1234
Refs GH-1234
```

- `Fixes` closes the issue when merged.
- `Refs` links without closing.
- Do not invent issue IDs.

## Attribution

Do not add agent, tool, or AI attribution to commit messages.

Avoid:

- `Co-Authored-By` footers for generated changes
- `Generated by AI`
- `Written with Claude`
- tool names in the subject, body, or footer

## Command Hygiene

Prefer separate `-m` arguments for body/footer text:

```bash
git commit -m "fix(api): Handle null response" \
  -m "Return a typed error when the upstream API returns null."
```

Use `git commit` with the editor when the message needs careful wrapping.

## Examples

```text
fix(api): Handle null response

Return a typed error when the upstream API returns null so callers can
surface a recoverable failure instead of crashing.

Fixes #1234
```

```text
feat(api)!: Remove deprecated v1 endpoint

Remove the v1 endpoint after its replacement has been available for one
release cycle.

BREAKING CHANGE: v1 endpoint requests now return 410.
```

```text
revert: feat(api): Add bulk import endpoint

This reverts commit abc1234.

Reason: The endpoint caused request timeouts on large imports.
```
